# External Integrations

**Analysis Date:** 2026-07-07

## APIs & External Services

**Nostr Relay Network (WebSocket):**
- Nostr relays ‚Äî Core transport for all Nostr events (messages, zaps, replies, group posts, profiles, mute lists, mailbox lists)
  - SDK/Client: `applesauce-relay` `RelayPool` ‚Äî singleton at `services/nostr.ts:60`
  - Connection: WebSocket (`wss://`), no auth token; per-relay NIP-42 authentication handled via `NostrConnectSigner` (`services/nostr.ts:146-186`)
  - Configurable relay sets:
    - Lookup relays ‚Äî from `config.json` `lookupRelays` (default `["wss://purplepag.es"]` in `const.ts:7`); used by event loader for profile/mailbox discovery
    - User outbox/inbox relays ‚Äî loaded dynamically from NIP-65 mailbox list (`services/nostr.ts:86-97`)
    - Direct message relays ‚Äî loaded from `kinds.DirectMessageRelaysList` (`services/nostr.ts:100-119`)
    - Signer relay ‚Äî hardcoded default `wss://relay.nsec.app` (`const.ts:8`); used for NIP-46 Nostr Connect signing
  - Subscription patterns: `pool.subscription(relays, filter, opts)` for live event streams; `pool.relay(url).request(filter)` for one-shot queries (`notifications/groups.ts:72-82`)
  - Ping/reconnect enabled: `enablePing: true`, `onUnresponsive: () => "reconnect"` (`services/nostr.ts:60-63`)

**ntfy Push Notifications (HTTPS):**
- ntfy.sh (or self-hosted ntfy server) ‚Äî Delivers push notifications to mobile/desktop devices
  - SDK/Client: Native `fetch()` POST ‚Äî `services/ntfy.ts:198-202`
  - Endpoint: `${server}/${topic}` (POST with `text/plain` body, ntfy-specific `X-*` headers)
  - Auth: None ‚Äî topic name acts as the credential (random `nanoid` generated by default in `services/config.ts:55`)
  - Config: `server` (default `https://ntfy.sh` at `services/ntfy.ts:134`), `topic`, `email` (optional fallback) ‚Äî all from `config.json`
  - Features used: title, priority (1-5 enum `NtfyPriority`), tags, click URL, icon, email, delay, markdown, actions (`services/ntfy.ts:43-74`)
  - Called from: all notification modules (`notifications/messages.ts:145`, `notifications/zaps.ts:107`, `notifications/groups.ts:131`, `notifications/replies.ts`)

**QR Server API (HTTPS, third-party):**
- `api.qrserver.com` ‚Äî Generates QR code images for Nostr Connect URIs and ntfy subscription links
  - SDK/Client: Direct image URL embedded in `<img src>` ‚Äî no fetch from server
  - Endpoint: `https://api.qrserver.com/v1/create-qr-code/?size=...&data=...`
  - Auth: None
  - Usage: `pages/signer.tsx:36`, `pages/home.tsx:259`, `pages/mobile.tsx:14`

**Chachi Chat (external link template only):**
- `chachi.chat` ‚Äî Group notification click-through links
  - Not an API integration ‚Äî only a URL template in `const.ts:9`: `https://chachi.chat/{hostname}/{group}`
  - Configurable via `config.json` `groups.groupLink`

## Data Storage

**Databases:**
- None ‚Äî No persistent database. All Nostr event data is in-memory only via `EventStore` singleton (`services/nostr.ts:59`)

**File Storage:**
- Local filesystem only ‚Äî Single JSON config file at `Bun.env.CONFIG ?? "config.json"` (`services/config.ts:84`)
  - Read on startup, auto-written on every config change via RxJS subscription (`services/config.ts:108-110`)
  - In Docker: mounted volume at `/app/data/config.json` (`Dockerfile:9-11`)
  - Config type: `AppConfig` in `services/config.ts:7-52`

**Caching:**
- In-process RxJS caching only ‚Äî `shareReplay(1)`, `shareAndHold()` (custom operator at `services/nostr.ts:52-57`) for mailboxes, message inboxes, mute lists, group list, people lists
- In-memory log buffer ‚Äî capped at 10,000 entries in `services/logs.ts:9`

## Authentication & Identity

**Nostr Identity (NIP-46 Nostr Connect):**
- Remote signer service (Alby, nsec.app, Amber, or any NIP-46 compatible app)
  - Implementation: `NostrConnectSigner` from `applesauce-signers` (`const.ts:2`, `pages/signer.tsx:3`)
  - Connection methods: QR code scan via `getNostrConnectURI()` (`pages/signer.tsx:33`), or manual bunker URI entry via `NostrConnectSigner.fromBunkerURI()` (`pages/signer.tsx:312`)
  - Signer relay: `wss://relay.nsec.app` default (`const.ts:8`)
  - Signing permissions: `NostrConnectSigner.buildSigningPermissions([kinds.ClientAuth])` (`const.ts:11-13`)
  - Persistence: Signer serialized to `config.json` as `SerializedAccount` via `account.toJSON()` (`pages/signer.tsx:276`); restored via `NostrConnectAccount.fromJSON()` (`services/nostr.ts:138`)
  - Required for: decrypting direct messages (NIP-04 and NIP-17 gift wraps), authenticating to relays (NIP-42)
  - Optional for: replies and zaps notifications (which don't require decryption)

**Relay Authentication (NIP-42):**
- Automatic AUTH to relays requiring it ‚Äî `services/nostr.ts:146-186`
  - Triggered when `relay.authRequiredForRead$` emits true
  - Uses signer to sign AUTH event via `relay.authenticate(signer)`

**Web UI Authentication:**
- None ‚Äî No authentication on the HTTP server; anyone with network access to port 8080 can view/modify config

## Monitoring & Observability

**Error Tracking:**
- None ‚Äî No external error tracking service

**Logs:**
- Custom in-memory log buffer ‚Äî `services/logs.ts`
  - `log(message, details?)` function pushes to module-level `logs` array (max 10,000 entries, FIFO pruning)
  - Also writes to `console.log`
  - Exposed via `/status` page (`pages/status.tsx`)
- Global error handlers ‚Äî `process.on("unhandledRejection")` and `process.on("uncaughtException")` in `index.ts:65-69` log to stderr but do not exit

## CI/CD & Deployment

**Hosting:**
- Docker container published to GitHub Container Registry (`ghcr.io`)
- Designed to run as a long-lived process (Bun HTTP server + Nostr relay subscriptions)

**CI Pipeline:**
- GitHub Actions ‚Äî `.github/workflows/docker.yml`
  - Triggers: push to any branch, semver tags (`v*.*.*`)
  - Steps: checkout ‚Üí QEMU ‚Üí Docker Buildx ‚Üí login to `ghcr.io` (using `GITHUB_TOKEN` secret) ‚Üí metadata extraction ‚Üí build & push multi-platform image (`linux/amd64,linux/arm64`)
  - Image tags: semver version, major.minor, major, branch name, PR ref, git SHA

## Environment Configuration

**Required env vars:**
- `PORT` ‚Äî HTTP listen port (optional, default `8080`)
- `CONFIG` ‚Äî Path to config JSON (optional, default `config.json`; set to `/app/data/config.json` in Docker)
- `NODE_ENV` ‚Äî Set to `production` in Docker image

**Required config.json fields (for full functionality):**
- `pubkey` ‚Äî Hex pubkey of the user to monitor (required for all notifications)
- `lookupRelays` ‚Äî Relays for profile/mailbox discovery (defaults provided)
- `topic` ‚Äî ntfy topic name (auto-generated random `nanoid` on first run)
- `server` ‚Äî ntfy server URL (defaults to `https://ntfy.sh`)
- `signer` ‚Äî Serialized NIP-46 account (required for DM decryption and relay auth)
- Per-category enabled flags and whitelist/blacklist arrays

**Secrets location:**
- No `.env` files (gitignored)
- No secrets in environment variables ‚Äî ntfy topic name is the only "secret" and is stored in `config.json` (gitignored at `/config.json` in `.gitignore:37`)
- GitHub Actions uses built-in `GITHUB_TOKEN` for container registry auth (`.github/workflows/docker.yml:36`)

## Webhooks & Callbacks

**Incoming:**
- None ‚Äî No webhook endpoints. The HTTP server only serves HTML pages, SSE streams, and static CSS

**Outgoing:**
- ntfy push notifications ‚Äî POST to `${ntfyServer}/${topic}` for each notification event (`services/ntfy.ts:198`)
- Nostr event publishing ‚Äî via `RelayPool.publish()` (configured as `NostrConnectSigner.publishMethod` at `services/nostr.ts:67`); used for NIP-42 AUTH events and signer handshake
