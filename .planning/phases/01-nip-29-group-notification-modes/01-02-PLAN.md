<objective>
Add persistent storage for per-group notification modes to AppConfig and make it
survive the existing installed base. The riskiest part is not the new field — it
is the boot-time loader's SHALLOW `{ ...config$.value, ...parsed }` merge, which
replaces the whole `groups` object from disk and would leave every pre-phase
install with `groups.modes === undefined`. This plan adds the field, the default,
and the mandatory backfill migration (mirroring the existing `messages`
migration), proves both round-trip and backfill with `bun:test`, and records the
D-07 behavior-change decision.

Purpose: The notification gate (Plan 03) and both pages (Plans 04, 05) read and
write `config.groups.modes`. This plan guarantees that field always exists as an
object, for new and existing installs alike.

Output: `services/config.ts` with the `modes` field + default + migration; a
config test with a pre-modes fixture; and a CHANGELOG entry for D-07.

Decision (D-07 — resolves RESEARCH Open Question 1): Phase 1 surfaces the
quieter-by-default behavior change via CHANGELOG.md ONLY. No in-app migration
banner and no new `AppConfig` flag (e.g. `migrationNoticeShown`) are added — that
is deferred as unneeded state per RESEARCH's recommendation. The optional
`/groups` banner described in 01-UI-SPEC.md is explicitly NOT built this phase.
</objective>
