<objective>
Create the two pure, network-safe units this phase's hardening is built from,
each with its test, mirroring Phase 3's extracted `notifications/legacy-messages.ts`
precedent (D4-09):

1. `helpers/gift-wrap-subscription.ts` — `notifyNewGiftWraps(seed$, live$)`, the
   seed-then-live seen-by-id dedup combinator that is the heart of the D4-02 fix
   (replacing the fragile limit-1 + skip-one "only new ones" hack). It emits only
   genuinely-new gift wraps: the historical backlog fetched during the seed phase is
   recorded but never emitted, and the live feed is filtered against the same
   growing `seen` set on every emission (Pitfall 1).
2. `notifications/gift-wrap-messages.ts` — `unlockPrivateDirectMessage(event, signer)`,
   the unwrap-and-classify unit: it calls applesauce's already-correct
   `unlockGiftWrap` (D4-01: the decrypt shape is already right) and returns the inner
   rumor only when its kind is PrivateDirectMessage (kind 14), else undefined.

Both units have zero top-level singleton imports so their tests import them directly
without booting the live pool/store. Wave 2 (Plan 02) wires them into
`services/nostr.ts` and `notifications/messages.ts`.

Purpose: The correctness bug (D4-02) and the testability goal (D4-09) both depend on
factoring the decision logic out of the self-subscribing modules. Getting these two
pure units and their deterministic tests green first means Wave 2 is pure wiring with
a full-suite safety net already in place.
Output: two new source modules, two new network-safe test files, all green.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| inbound gift-wrap event (relay) → seed$/live$ streams | Untrusted event ids/content enter the dedup combinator; the combinator must classify new-vs-seen by id alone (never the randomized created_at). |
| inbound gift-wrap event (relay) → unlockGiftWrap (JSON.parse of seal/rumor) | A malformed/malicious wrap can throw inside applesauce's unwrap; this unit surfaces the throw to its caller (never hand-rolls the decrypt). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-4-01 | Denial of Service (availability) | notifyNewGiftWraps combinator | medium | mitigate | The seed(ignoreElements)+live(seen-filter) design guarantees a genuinely-new wrap is never dropped and a historical wrap is never re-notified; `seen` is checked on every live emission (Pitfall 1). Covered by the 4 dedup-contract tests incl. the zero-historical edge case that broke skip-one. Residual: low. |
| T-4-02 | Spoofing/Tampering | unlockPrivateDirectMessage kind classification | low | mitigate | Returns undefined for any rumor kind ≠ 14, so a non-DM rumor (e.g. group-chat) cannot reach the DM-notification path. Covered by the non-DM-kind test case. |
| T-4-03 | Denial of Service (input validation, V5) | unlockGiftWrap JSON.parse of a malformed wrap | low | mitigate | The unit deliberately does NOT swallow a failed unwrap — a throw propagates to the caller's catch(EMPTY) isolation (Plan 02, D4-05). Documented so the caller-side isolation is not accidentally duplicated here. Crypto (V6): always via applesauce unlockGiftWrap, never hand-rolled NIP-44. |

No new packages are installed this phase (RESEARCH §Package Legitimacy Audit: "Not applicable"), so there is no supply-chain (T-*-SC) threat and no install checkpoint. Residual after mitigation: low across the board; no unmitigated high-severity residual — no manual-review flag required.
