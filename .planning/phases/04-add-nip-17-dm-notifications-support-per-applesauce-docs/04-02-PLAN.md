<objective>
Wire the two Wave-1 pure units into the live pipeline, completing the NIP-17
review/harden (D4-01: harden, do not rebuild — unlockGiftWrap and the rumor-kind
classification were already the documented-correct shape). Two files change:

1. services/nostr.ts — the centerpiece D4-02 fix: rewrite giftWraps$ from the fragile
   limit-1 + one-event-drop "only new ones" hack to a seed-then-live composition. The
   seed is a one-shot pool.request() that completes on EOSE and persists the current
   backlog into the shared eventStore (never emitting it downstream); the live phase is
   the existing persistent pool.subscription({ reconnect: Infinity }). The new
   notifyNewGiftWraps combinator (Plan 01) sequences them so only genuinely-new gift
   wraps reach subscribers — never dropping a real DM, never re-notifying history, never
   relying on the randomized created_at. The now-dead one-event-drop rxjs import is
   removed (Pitfall 2).
2. notifications/messages.ts — the NIP-17 block only: rewire to the extracted
   unlockPrivateDirectMessage + defined() (D4-09); apply the safe
   `error instanceof Error ? error.message : String(error)` guard to the catchError log
   (D4-06); add click:buildOpenLink(rumor as unknown as NostrEvent) so DM notifications
   deep-link like replies/zaps and the NIP-04 block (D4-04); and, as the RESEARCH-
   recommended discretionary parity fix (Open Question 1 / Pitfall 4, within the D4-08
   NIP-17 boundary), guard the profile lookup with .catch(() => undefined) and the
   getMessageDisplayName fallback so a profile-lookup timeout can never throw into the
   subscribe callback or render a literal "undefined" title. Confirm — do not change —
   the generic title + sendContent-gated body (D4-07), and deliberately add NO reconnect
   hint (D4-05). No const.ts change (D4-03).

Purpose: Close the one real correctness bug (genuinely-new DMs can currently be dropped
by the skip-based hack) and bring the NIP-17 notification to parity with the NIP-04 path
hardened in Phase 3 — reusing only already-exported, already-documented applesauce
primitives plus the Wave-1 units, against the full-suite safety net Plan 01 established.
Output: rewritten giftWraps$ in services/nostr.ts and a hardened NIP-17 block in
notifications/messages.ts; full suite + lint green.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| inbound gift-wrap (relay) → decrypt pipeline | Untrusted, possibly malformed/undecryptable/spam wrap enters the NIP-17 mergeMap. |
| decrypted DM plaintext → ntfy server | When messages.sendContent is true, plaintext crosses to a third-party server; the title never carries content. |
| decrypt error object → logs | Error objects may embed ciphertext/plaintext fragments; only the message string is logged. |
| rumor → click deep-link (appLink template) | buildOpenLink substitutes an nevent (from the rumor's id/kind/pubkey) into the operator-controlled appLink template. |
| seed pool.request() → shared eventStore | The historical gift-wrap backlog is persisted into the single in-memory EventStore. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-4-04 | Information Disclosure | decrypted DM body → ntfy | high | mitigate | Body is plaintext only when messages.sendContent is true (default false; the Phase-3 migration keeps upgraders off), else "[content omitted]"; the title stays generic (D4-07 — confirm, no regress). Residual: low (explicit user opt-in). |
| T-4-05 | Denial of Service | NIP-17 mergeMap on a malformed/spam wrap | medium | mitigate | The existing catchError(() => EMPTY) is preserved — one undecryptable/rejected wrap cannot tear down the messages subscription (D4-05/D4-06). Covered by Plan 01's non-DM unit test + the full-suite gate. |
