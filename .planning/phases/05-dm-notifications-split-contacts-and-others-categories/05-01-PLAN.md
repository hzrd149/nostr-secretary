   notifications/legacy-messages.ts's extract-for-testability precedent, plus its unit test.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| inbound kind-3 event (relay) -> contacts$ classification | An untrusted, possibly malformed follow-list event enters eventStore.contacts()/getContacts() and feeds isContact. |
| slow/unreachable follow-list relay -> isContact resolution | The follow list may never load; isContact must still resolve deterministically without blocking the caller. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-5-01 | Tampering | malformed kind-3 event -> getContacts parse | low | mitigate | applesauce's getPublicContacts filters to well-formed p-tags only (existing library behavior, unchanged); isContact reads ProfilePointer[].some(...) which is total over any list -- a garbage event yields an empty/partial list, never a throw. |
| T-5-02 | Denial of Service (availability) | isContact awaiting a follow list that never loads | low | mitigate | isContact wraps contacts$ in timeout({first:2000, with:()=>of([])}) (isMuted's proven idiom): a slow relay resolves false (=> "others") within 2s and can never block or crash the caller (D5-02). Residual: a not-yet-loaded follow list transiently classifies a real contact as "others" -- accepted, self-heals on the next DM once the list loads (D5-03), and "others" defaults ON for new installs so no DM is silently dropped by default. |

No new packages are installed this plan (RESEARCH Package Legitimacy Audit: "Not
applicable"), so there is no supply-chain (T-*-SC) threat and no install checkpoint. Both
threats are low-severity and mitigated -- no unmitigated high-severity residual, so no
manual-review flag is required (ASVS L1, block-on-high).
