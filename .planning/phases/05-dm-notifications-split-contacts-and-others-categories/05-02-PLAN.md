<objective>
The riskiest, most cross-cutting piece of Phase 5: the messages config-schema cutover.
`messages.enabled` is read/written in 6 production sites plus 2 test files, and this plan
replaces it with nested per-category `messages.contacts.enabled` / `messages.others.enabled`
everywhere at once -- a single whole-project `tsc` atomic refactor -- while (a) defaulting
NEW installs to contacts-on / others-off (D5-05, corrected), (b) preserving existing users'
behavior on upgrade (D5-06), (c) keeping enabled$ gating working (D5-04), (d) extending the
Phase-2 kind-30078 sync to carry both flags, and (e) NOT silently disabling both categories
when a not-yet-upgraded peer device's old-schema payload arrives (D5-10 / Pitfall 5).

Files: services/config.ts (schema + seed + migration), helpers/preferences.ts (sync +
version bump + old-schema fallback), notifications/messages.ts (the enabled$ one-liner
ONLY), pages/messages.tsx (two-section UI), and both existing test files (migrated to the
new shape + new regression cases). This plan does NOT wire the category gate into the DM
listeners -- that is Plan 03, layered on top of this schema in Wave 2.

Purpose: land the schema, defaults, migration, sync, and UI as one coherent, typecheck-green
cutover so Plan 03 can add the behavioral gate against a stable, tested config surface.
Output: the reshaped config + sync + two-section UI, all readers migrated, full regression
coverage; `bun run lint` and full `bun test` green.
</objective>
