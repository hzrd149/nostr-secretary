<objective>
Add the minimal rate-limit UI (D6-08): a single "0 = unlimited" number input on each of the four
per-type config pages (bound to config.rateLimit.perType.<type>) and, on pages/notifications.tsx,
a global-limit input plus a window input -- which requires adding that page's FIRST-EVER PATCH route
(it is GET-only today, RESEARCH Pitfall 7). Reuse the existing Datastar form + flat-signal +
ServerSentEventGenerator PATCH pattern from pages/replies.tsx, clamping every incoming value to a
non-negative integer server-side (ASVS V5) and spreading the rest of currentConfig.rateLimit so a
single-field edit never clobbers sibling limits.

This is a Wave-2 plan consuming Plan 02's config.rateLimit shape. It touches only page files --
disjoint from the limiter service/listener plan (Plan 03) -- so both Wave-2 plans run in parallel.

Purpose: give users a discoverable, minimal control over each per-type limit, the global limit, and
the window, with the 0=unlimited escape hatch visible (D6-08/D6-09). Output: five modified pages
(four per-type inputs + notifications.tsx's global/window inputs behind a new PATCH route); a
per-task human-check (recorded as UAT for /gsd-verify-work) confirms the fields render, save, and
persist; `bun run lint` clean.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PATCH request signals -> page handler -> config$ | Each page's PATCH reads client-supplied Datastar signals (the number input values) and merges them into the app config; a negative/non-numeric/oversized value is untrusted input crossing into config.rateLimit. |
| new mutating surface on pages/notifications.tsx | This page had no write path before; adding a PATCH introduces the first config-mutating entry point on it. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-6-11 | Tampering | a PATCH submitting a negative / non-numeric / NaN rate-limit value corrupting config.rateLimit arithmetic downstream (evaluate's window math) | high | mitigate | Each PATCH handler clamps its incoming signal to a non-negative integer (finite >= 0 else current/default, floors floats) before config$.next, mirroring pages/replies.tsx's existing whitelist/blacklist sanitization and Plan 02's asNonNegativeInt discipline (ASVS V5). Residual: low. |
| T-6-12 | Tampering | a single-field save clobbering sibling rateLimit limits (e.g. saving a per-type value wiping global/window or the other per-type entries) | medium | mitigate | Every merge spreads currentConfig.rateLimit (and currentConfig.rateLimit.perType for per-type pages) so only the edited field changes; verified by the sibling-preservation behavior check + the human-verify step 4. Residual: low. |
| T-6-SC | Tampering | npm/pip/cargo installs | n/a | n/a | No packages installed this plan (RESEARCH Package Legitimacy Audit intentionally empty) -- no supply-chain surface, no install checkpoint. |

The high-severity threat (T-6-11 untrusted-number coercion at the new write surface) is fully
mitigated by the per-handler non-negative-integer clamp, consistent with the existing page
sanitization pattern and Plan 02's sync-side coercion -- no unmitigated high-severity residual, no
manual-review flag required (ASVS L1, block-on-high).
</threat_model>
