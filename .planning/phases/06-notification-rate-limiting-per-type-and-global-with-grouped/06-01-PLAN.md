<objective>
Create the pure, clock-injectable accounting/summary unit that is the deterministic core of
Phase 6's rate limiter. `services/rate-limit-accounting.ts` holds ALL window math, the
deliver-vs-accumulate decision, the combined-summary formatting, and the window reset -- with
zero I/O and zero timers, taking `now: number` as an explicit parameter everywhere. It mirrors
the Phase 3-5 precedent of an extracted pure unit (`notifications/dm-notification-gate.ts`) that
tests import directly without touching the self-subscribing service singletons.

The impure shell (`services/rate-limit.ts`, Plan 03) and the config/sync (Plan 02) are built on
top of the types and functions defined here. This plan is a foundation with no dependencies.

Purpose: land the tumbling-window accounting logic as a fully unit-tested pure module so the
riskier wiring (RxJS timer, the 5 call-site swaps) in Wave 2 rests on a proven core, and so the
security-critical counts-only guarantee (D6-10) is enforced structurally by the type shape.
Output: `services/rate-limit-accounting.ts` + its full clock-injected test matrix; `bun run lint`
and full `bun test` green.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| accumulated per-type overflow -> grouped summary string | The overflow accounting is the sole source of the combined summary body; if it could hold message content, DM plaintext could leak into a summary regardless of sendContent. |
| injected config numbers -> window arithmetic | evaluate/flushOverflow consume a RateLimitConfig whose fields originate (in Plan 02/03) from config + a decrypted sync payload; the pure unit must behave sanely for edge values (0=unlimited). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-6-01 | Information Disclosure | grouped summary carrying DM plaintext instead of counts (D6-10) | high | mitigate | RateLimitState.overflow is Record<NotificationType, number> -- structurally incapable of holding an NtfyNotificationOptions or message string; formatOverflowSummary only ever emits "<integer> <static label>" parts. Asserted by exact-string tests over known counts. Residual: low (a future widening of the type would be required to regress, and is prohibited + review-gated). |
| T-6-02 | Denial of Service (availability) | a qualifying over-limit notification being dropped without accounting (D6-04) | medium | mitigate | evaluate() increments overflow[type] on every non-delivered notification and never returns without either delivering or accumulating; covered by the over-per-type and over-global accumulation tests. Residual: low. |
| T-6-03 | Tampering | pathological config values (0, very large) producing negative/unbounded arithmetic in evaluate | low | mitigate | 0 is handled explicitly as "unlimited" (gate skipped); counts are monotonic increments of finite integers within a window that resets each flush; no subtraction/division on config values. Coercion of untrusted numeric config happens upstream in Plan 02 (asNonNegativeInt). Residual: low. |
| T-6-SC | Tampering | npm/pip/cargo installs | n/a | n/a | No packages installed this plan (RESEARCH Package Legitimacy Audit: "table intentionally empty") -- no supply-chain surface, no install checkpoint. |

No unmitigated high-severity residual: T-6-01 (counts-only) is mitigated structurally by the
Record<...,number> type and exact-string tests -- no manual-review flag required (ASVS L1,
block-on-high).
