<objective>
Wire the rate limiter into the notification path. Create the impure choke-point service
`services/rate-limit.ts` -- `rateLimitedNotify(type, options)` that consults the pure tumbling-window
accounting unit (Plan 01) against the synced/migrated `config.rateLimit` (Plan 02), delivering via
`sendNotification` when under limit and accumulating per-type overflow when over -- plus the
config-driven RxJS flush timer that emits ONE combined counts-only grouped summary per window,
bypassing the limiter (D6-01/D6-05/D6-06). Then swap all 5 existing `sendNotification(...)` call
sites (replies, zaps, both DM sites, groups) to route through it 1:1, leaving the existing
`shouldNotify`/category gates exactly where they are (D6-03/D6-10).

This is a Wave-2 plan: it consumes Plan 01's accounting API (`evaluate`/`flushOverflow`/
`formatOverflowSummary`/`createRateLimitState`/`NotificationType`) and Plan 02's `config.rateLimit`
shape (`configValue("rateLimit")`/`getConfig().rateLimit`). It touches only the new service, its
test, and the four listener files -- disjoint from the UI plan (Plan 04), so both Wave-2 plans run
in parallel.

Purpose: turn the proven accounting core into a live, single-choke-point limiter that is the last
gate before delivery and never lets the grouped summary leak DM content or suppress itself.
Output: `services/rate-limit.ts` + its bypass/accumulate test + the 5 call-site swaps; `bun run lint`
and full `bun test` green.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| accumulated per-type overflow -> grouped summary body | The flush turns per-type integer counts into the summary message; if message content ever entered the rate state, DM plaintext could leak into the summary regardless of sendContent. |
| listener call site -> rateLimitedNotify | The swap is the seam where "qualifies for a notification" (decided by the untouched shouldNotify + category gates) meets "delivered now vs. grouped"; a sloppy swap could accidentally move or drop a gate. |
| live config.rateLimit -> flush timer | A live window edit restarts the interval via switchMap; pending overflow must not silently vanish (RESEARCH Pitfall 8, hardening). |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-6-07 | Information Disclosure | grouped summary carrying DM plaintext instead of counts (D6-10) | high | mitigate | The summary body is produced ONLY by formatOverflowSummary over Record<NotificationType,number> counts (Plan 01, proven counts-only); rate-limit.ts never stores or reads options.message into the overflow/summary path; the flush test asserts the emitted body equals the counts-only summary. Residual: low (regression would require widening the accounting overflow type, prohibited + review-gated in Plan 01). |
| T-6-08 | Tampering / Elevation | a call-site swap accidentally bypassing or reordering shouldNotify / the Phase-5 category gate (changing which events qualify) | high | mitigate | The swap changes ONLY the final call target + a type argument; the gates above each send are unchanged (verified by review + the positive-grep gate + full `bun test` regression). Residual: low. |
| T-6-09 | Denial of Service (availability) | the grouped summary being routed back through rateLimitedNotify and thus suppressed when the bucket is saturated (D6-06) | medium | mitigate | The flush calls sendNotification directly; the test drives the flush while the per-type bucket is saturated and asserts the injected send still fires exactly once. Residual: low. |
