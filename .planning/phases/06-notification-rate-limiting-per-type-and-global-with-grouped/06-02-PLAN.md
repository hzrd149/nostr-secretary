<objective>
Land the persistence + sync + migration surface for rate limiting: add `AppConfig.rateLimit`
(global + per-type + window) with anti-spam defaults, a defensive `migrateConfig` backfill, and
extend the Phase-2 kind-30078 sync (`SyncedPrefs.rateLimit`, `PREFS_VERSION` 2->3,
serialize/sanitize/merge) -- with the one non-obvious twist this phase's research flags as its
riskiest correctness call: an inbound payload that has NO `rateLimit` key (a not-yet-upgraded
peer) must fall back to this device's own safe local defaults, NEVER to 0/unlimited (RESEARCH
Pitfall 6 -- the inverse of Phase 5's `asMessagesCategories` legacy-seed pattern, because rate
limiting has no legacy predecessor field to seed from).

This plan is a Wave-1 foundation, independent of Plan 01 (it defines the config shape inline
rather than importing the accounting module's `RateLimitConfig`, so the two Wave-1 plans stay
parallel with disjoint files). Plans 03 (call-site swaps + the impure limiter) and 04 (UI) build
on the `config.rateLimit` field this plan creates.

Purpose: give the limiter a synced, migrated, defaulted config surface with untrusted-input
coercion, and document the additive default behavior change (D6-09). Output: the reshaped config +
migration + sync + a CHANGELOG entry + full regression coverage; `bun run lint` and full `bun test`
green.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| inbound decrypted kind-30078 payload -> sanitizeSyncedPrefs -> config | An untrusted synced payload (another device or an interoperating app) sets the rate-limit numbers; a pre-Phase-6 payload with no rateLimit key is a first-class case. |
| stored config.json -> migrateConfig -> config$ | A hand-edited or older config.json may carry an absent/partial/malformed rateLimit. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-6-04 | Denial of Service (availability) | a pre-Phase-6 peer's payload (no rateLimit key) silently DISABLING rate limiting (falling back to 0/unlimited) on an upgraded device -- or, inverted, silently over-suppressing | high | mitigate | sanitizeSyncedPrefs falls back to a structuredClone of the device's own DEFAULT_RATE_LIMIT_CONFIG when rateLimit is absent (RESEARCH Pitfall 6), never to 0/unlimited; PREFS_VERSION bumped to 3; an explicit regression test asserts the absent-key result deep-equals the defaults. Residual: low. |
| T-6-05 | Tampering | a malformed/hostile decrypted rateLimit (negative, NaN, non-numeric, "DROP TABLE") corrupting config arithmetic | high | mitigate | asNonNegativeInt coerces every numeric field (finite && >=0 else per-field local default; floors floats), mirroring the existing asStringArray/asBoolean ASVS V5 discipline -- a malformed payload can never inject a non-numeric/negative into config.rateLimit. Migration applies the same defensive backfill to stored config. Residual: low. |
| T-6-06 | Tampering | a compromised signer publishing rateLimit.global:999999999 to effectively disable limiting via sync | low | accept | Not a NEW surface: the same signer compromise already lets an attacker rewrite whitelists/blacklists via the same kind-30078 mechanism; the existing D2-08 high-water-mark (isNewerPrefs) + self-encryption bound it. No new mitigation required (RESEARCH Security Domain, flagged for completeness). |
| T-6-SC | Tampering | npm/pip/cargo installs | n/a | n/a | No packages installed this plan (RESEARCH Package Legitimacy Audit: empty) -- no supply-chain surface, no install checkpoint. |

Both high-severity threats (T-6-04 sync fallback direction, T-6-05 untrusted-number coercion) are
fully mitigated by the local-defaults fallback + asNonNegativeInt coercion, proven by the explicit
absent-key regression + the coercer unit cases -- no unmitigated high-severity residual, no
manual-review flag required (ASVS L1, block-on-high).
</threat_model>
