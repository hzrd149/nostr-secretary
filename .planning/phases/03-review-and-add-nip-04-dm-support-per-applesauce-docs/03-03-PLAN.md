<objective>
Harden the existing NIP-04 legacy-DM listener in notifications/messages.ts so it
matches the robustness the NIP-17 path already has, plus close the remaining
review gaps (D3-01: harden, do not rebuild — the unlockLegacyMessage call is
already correct). Four in-scope changes to the NIP-04 block: (1) wrap its async
body (profile fetch + unlockLegacyMessage) in catchError(()=>EMPTY) so one bad DM
can't kill the subscription (D3-08); (2) flip a new exported nip04DecryptDegraded$
signal on decrypt failure / reset it on success, and render a non-blocking
"reconnect your signer" hint on /notifications (D3-07); (3) thread the raw event
through and add click:buildOpenLink(event) so DM notifications deep-link like
replies/zaps do (D3-06); (4) remove the dead deprecated correspondent-alias
import (D3-08). Confirm — don't change — the generic title + sendContent-gated
body (D3-05). Add the missing test coverage (D3-09). Stay strictly inside the
NIP-04 scope: the NIP-17 block and a shouldNotify refactor are out (D3-10).

Purpose: A single undecryptable DM currently risks tearing down all message
notifications, an already-connected signer that lacks the new permission gives no
feedback, and DM notifications aren't tappable. This plan fixes all three by
mirroring patterns already proven elsewhere in this same codebase.
Output: hardened notifications/messages.ts, a reconnect hint on /notifications,
and tests/notifications/messages.test.ts.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| inbound kind-4 event (relay) → decrypt pipeline | Untrusted, possibly malformed/undecryptable ciphertext enters the mergeMap. |
| decrypted DM plaintext → ntfy server | When sendContent is true, plaintext crosses to a third-party server; title never carries content. |
| decrypt error object → logs | Error objects may embed sensitive fragments; only the message string is logged. |
| event → click deep-link (appLink template) | buildOpenLink substitutes an nevent into the operator-controlled appLink template. |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-3-05 | Denial of Service | NIP-04 mergeMap (messages.ts) | medium | mitigate | Wrap the async body in catchError(()=>EMPTY) so one undecryptable/rejected/timed-out kind-4 event cannot tear down the whole messages subscription — parity with the NIP-17 path (D3-08). Test 1's garbage-ciphertext case documents the absorbed failure. |
| T-3-06 | Information Disclosure | decrypted DM body → ntfy | high | mitigate | Title stays generic (no content); body is plaintext only when sendContent is true (off by default + Plan 03-02 migration keeps upgraders off), else "[content omitted]" (D3-05). Residual: low (user-controlled explicit opt-in). |
| T-3-07 | Information Disclosure | catchError log | low | mitigate | Log only Reflect.get(error,"message") — never the raw error, event content, or ciphertext (matches NIP-17 block). Prohibition asserted in must_haves (D3-08). |
| T-3-08 | Tampering (soft DoS) | bunker grants < requested | low | mitigate | Non-blocking DmDecryptHint prompts reconnect on any decrypt failure while connected (D3-07). UX mitigation only — no cryptographic guarantee a bunker honors the grant (Pitfall 2). |
| T-3-09 | Information Disclosure | buildOpenLink deep-link (appLink template) | low | accept | Reuses the existing, already-shipped buildOpenLink used by replies/zaps; for kind-4 it encodes an nevent (event id + relay hints, NOT plaintext). No new leak surface; appLink is operator-controlled config. Raw-template-replace fragility is a pre-existing CONCERNS.md item, out of scope here (D3-10). |

Residual after mitigation: low across the board. The one high-severity threat
(T-3-06) is mitigated to a low residual by the existing sendContent gate plus the
Plan 03-02 migration fix; no unmitigated high-severity residual — no manual-review
flag required.
</threat_model>
