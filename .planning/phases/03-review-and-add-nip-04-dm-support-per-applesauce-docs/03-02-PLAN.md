<objective>
Close the privacy hole flagged in CONCERNS.md and D3-04: the config migration at
services/config.ts:111-119 sets `sendContent: parsed.directMessageNotifications`,
so any user upgrading with DM notifications already enabled is silently switched
to sendContent:true — sending decrypted DM plaintext to the (often third-party)
ntfy server with no explicit opt-in. Flip that default to an unconditional false,
and make the migration unit-testable by extracting the inline load-time migration
block into an exported pure function migrateConfig(parsed), mirroring this
codebase's established "extract a pure helper so it can be tested off the
singleton" convention (helpers/preferences.ts, helpers/groups.ts).

This is a strictly local change to services/config.ts and its test (D3-10): only
sendContent's default changes behavior; the directMessageNotifications→messages
reshape and the Phase-1 groups.modes backfill are preserved exactly.

Purpose: sendContent is a privacy toggle. Defaulting upgraders to "on" leaks
plaintext DMs. Off-by-default with explicit opt-in is the required posture.
Output: services/config.ts with sendContent:false + extracted migrateConfig;
a regression test that fails if the leak is ever reintroduced.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| on-disk config.json (legacy, possibly hand-edited) → in-memory AppConfig | Untrusted/stale persisted shape is reshaped at load; migration decides privacy defaults. |
| in-memory AppConfig → ntfy server | When sendContent is true, decrypted DM plaintext crosses to a third-party server. The migration default gates whether upgraders start in that state. |
