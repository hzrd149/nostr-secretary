<objective>
Make the connected bunker actually authorize the new capabilities this phase
needs — signing kind 30078 and NIP-44 encrypt/decrypt. Today `SIGNER_PERMISSIONS`
(`const.ts:11-13`) requests only ClientAuth and, critically, is DEAD CODE: no
call site passes it, so every new signer connection today requests no explicit
permissions at all and relies entirely on the bunker's default policy (Pitfall 4).

This plan does the full fix D2-13 requires: (a) expand the constant, and (b) wire
it into all three signer-creation call sites (two `getNostrConnectURI` calls in
pages/signer.tsx, one in pages/home.tsx — each appears twice because the QR image
re-encodes the URI — and the `NostrConnectSigner.fromBunkerURI` manual-bunker
path in pages/signer.tsx). Without step (b), expanding the constant has zero
runtime effect.

Purpose: The Plan 03 publish/subscribe pipelines call signer.signEvent(kind 30078)
and signer.nip44.encrypt/decrypt. A freshly connected bunker must have been asked
for those permissions at connect time, or those calls prompt-on-every-use or fail.

Output: an expanded, tested `SIGNER_PERMISSIONS` plus three wired call sites.

Note on already-connected signers (Pitfall 4): applesauce-signers@6.0.1 exposes
no re-request-permissions API for an existing session. This plan requests the
right permissions for NEW connections; Plan 03 handles an under-permissioned
existing session gracefully (timeout + catch + local-only fallback). Do not
attempt a silent re-grant here — there is no SDK method for it.
</objective>
