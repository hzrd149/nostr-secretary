<objective>
Build the reactive singleton that makes settings sync real: publish-on-change and
subscribe-and-apply for the kind-30078 encrypted preferences event. This mirrors
the existing services/nostr.ts observables — groups$ (reactive replaceable read),
mutedPubkeys$ (decrypt-in-switchMap), and tagged$ (live pool.subscription) — using
the pure helpers proven in Plan 01 for serialization, validation, and conflict
resolution, and the expanded permissions wired in Plan 02 for signing/encryption.

Two pipelines, one file:
1. Publish: config$ → skip boot emission → serializePrefs → drop no-op changes →
   debounce ~1.5s → equality-guard against the last-known payload (loop prevention,
   D2-09) → if a signer is present, timeout-wrapped nip44-encrypt + signEvent →
   publish to outbox relays (fallback lookupRelays) → record the published payload.
2. Subscribe: a live pool.subscription REQ (reconnect:Infinity, resubscribe:true)
   feeding eventStore, plus a reactive eventStore.replaceable read; on each newer
   event, timeout-wrapped decrypt → sanitize → high-water-mark gate → mergePrefs →
   updateConfig, while recording the applied payload so the publish pipeline no-ops.

Purpose: This delivers the phase goal — preferences persist off-device, sync across
clients, are writable by other apps, and are subscribable by the notification
server. It degrades gracefully to today's local-only behavior when no signer is
connected (D2-12) and never loses a local save or hangs on a bad bunker
(D2-14/D2-15).

Design guardrails (hard constraints from RESEARCH):
- D2-14 premise correction: `signer.nip44` is NEVER undefined for the wired
  NostrConnectAccount, so do NOT gate on a null nip44. The real risk is a NIP-46
  round trip hanging forever (no built-in timeout). Wrap EVERY signer round trip
  in an ~8s timeout and treat timeout/rejection as "degrade to local-only + log".
- D2-09: use payload-equality (lastKnownPayloadJSON), NOT a skip(N) counter or a
  timing flag — a debounced pipeline defeats timing flags (Pitfall 5).
- config.ts is NOT modified: the save-on-change write always runs (never lose a
  local setting, D2-15); loop prevention lives entirely in this new service.
- Do NOT add a unit test that imports services/preferences.ts or services/nostr.ts
  (Pitfall 6 / WR-04): importing them opens live relay connections. This service's
  wiring is verified by lint + no-regression full suite + structural checks, and
  its live behavior is a documented manual verification (VALIDATION.md).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| app → bunker (NIP-46) | signEvent + nip44 encrypt/decrypt round trips over relays; can hang or reject |
| relay/other-app → app | inbound kind-30078 events (possibly spoofed, replayed, or malicious) decrypted and merged into config$ |
| app → relays | outbound publish of the encrypted prefs event to outbox relays |
| config plaintext → logs | decrypted prefs must not leak into the in-memory log buffer |
